Below is a proposed data model for ingesting and persisting your uncle’s daily A+ Trade Setups. You can implement this either as Pydantic schemas (in common/models.py aplus planning) or as a relational schema via SQLAlchemy/PostgreSQL. The goal is to capture every signal type, its parameters, and the daily bias rules.

1. Pydantic Schemas (common/models.py)

from __future__ import annotations
from enum import Enum
from datetime import date
from typing import List, Optional, Union
from pydantic import BaseModel

class ComparisonType(str, Enum):
    ABOVE = "above"
    BELOW = "below"
    NEAR  = "near"
    RANGE = "range"

class Aggressiveness(str, Enum):
    AGGRESSIVE   = "aggressive"
    CONSERVATIVE = "conservative"
    NONE         = "none"

class SignalCategory(str, Enum):
    BREAKOUT  = "breakout"   # “🔼 Breakout … Above”
    BREAKDOWN = "breakdown"  # “🔻 Breakdown … Below”
    REJECTION = "rejection"  # “❌ Rejection Near”
    BOUNCE    = "bounce"     # “🔄 Bounce From” or “🌀 Bounce Zone”

class Signal(BaseModel):
    category:       SignalCategory
    aggressiveness: Aggressiveness = Aggressiveness.NONE
    comparison:     ComparisonType
    trigger:        Union[float, List[float]]            # single price or [low, high] for RANGE
    targets:        List[float]                          # 1–3 target levels

class BiasFlip(BaseModel):
    comparison: ComparisonType    # e.g. BELOW
    price:      float             # flip price

class Bias(BaseModel):
    direction:    str              # "bullish" or "bearish"
    condition:    ComparisonType   # e.g. ABOVE
    price:        float             # bias reference price
    flip:         Optional[BiasFlip]

class TickerSetup(BaseModel):
    symbol: str
    signals: List[Signal]
    bias:    Optional[Bias]

class TradeSetupMessage(BaseModel):
    date:     date                  # e.g. 2025-05-14
    raw_text: str
    setups:   List[TickerSetup]

-- messages table
CREATE TABLE setup_messages (
  id        SERIAL PRIMARY KEY,
  setup_date DATE      NOT NULL,
  raw_text   TEXT      NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

-- one row per ticker in each message
CREATE TABLE ticker_setups (
  id               SERIAL PRIMARY KEY,
  message_id       INTEGER  REFERENCES setup_messages(id),
  symbol           TEXT     NOT NULL
);

-- signals for each ticker
CREATE TABLE signals (
  id               SERIAL PRIMARY KEY,
  ticker_setup_id  INTEGER   REFERENCES ticker_setups(id),
  category         TEXT      NOT NULL,
  aggressiveness   TEXT      NOT NULL,
  comparison       TEXT      NOT NULL,
  trigger_price    DOUBLE PRECISION,
  trigger_low      DOUBLE PRECISION,
  trigger_high     DOUBLE PRECISION
);

-- target levels per signal
CREATE TABLE signal_targets (
  id        SERIAL PRIMARY KEY,
  signal_id INTEGER REFERENCES signals(id),
  level     DOUBLE PRECISION NOT NULL,
  position  INTEGER          NOT NULL  -- 1,2,3 for target ordering
);

-- bias rules per ticker
CREATE TABLE biases (
  id               SERIAL PRIMARY KEY,
  ticker_setup_id  INTEGER   REFERENCES ticker_setups(id),
  direction        TEXT      NOT NULL,
  condition        TEXT      NOT NULL,
  price            DOUBLE PRECISION NOT NULL,
  flip_comparison  TEXT,
  flip_price       DOUBLE PRECISION
);
How to Use
Ingestion:

Create a TradeSetupMessage, store it in setup_messages.

For each TickerSetup, insert into ticker_setups and parse its Signal and Bias records.

Querying:

Quickly retrieve all SPY signals on a given date by joining setup_messages → ticker_setups → signals → signal_targets.

Bias rules live in their own table for easy lookups when evaluating exit conditions.

Extension:

If you later want to persist which options you chose against each signal, add an option_positions table linking to signals.id.

You can also version setups by adding a version or source column on setup_messages.
